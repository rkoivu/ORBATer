## Design Context

### Product Summary
ORBATer is a browser-based NATO APP-6 order of battle builder for creating military unit diagrams, command hierarchies, and force-structure maps. It is a working tool, not a marketing site. Users spend time building, editing, reorganizing, annotating, exporting, and sharing operational diagrams inside a dense, stateful workspace.

The product’s value comes from speed, clarity, and control:
- Build and edit ORBAT structures quickly
- Understand hierarchy and relationships at a glance
- Preserve, compare, and share versions without losing context
- Export work into formats that fit planning, briefing, and analysis workflows

### Users
Primary users inferred from the repository:
- Military planners
- Defense analysts
- Wargamers and scenario designers
- Researchers working with force-structure diagrams
- Staff users preparing briefing-ready diagrams under time pressure

### User Context
This interface is typically used in focused, task-heavy sessions rather than casual browsing. Users are likely to:
- Work on desktop or laptop in office, classroom, lab, or home-analysis settings
- Spend extended time in the app, not just a few minutes
- Need high-density information without visual clutter
- Switch between editing, reviewing, and presenting the same document
- Care more about legibility and confidence than novelty for its own sake

The app should feel trustworthy and operational rather than playful, lifestyle-oriented, or consumer-polished.

### Jobs To Be Done
Users come to ORBATer to:
- Create a clear hierarchy from scratch
- Import an existing structure and refine it
- Reorganize units fast while preserving metadata
- Compare snapshots and alternative versions
- Produce visuals suitable for internal planning, briefing, or sharing
- Understand command relationships and organizational depth quickly

### Emotional Goals
The interface should evoke:
- Control
- Clarity
- Confidence
- Readiness
- Composure under complexity

It should not evoke:
- Hype
- Gamified novelty
- Corporate SaaS blandness
- Futuristic sci-fi theatrics
- Decorative militarism

### Brand Personality
Three-word personality:
- Precise
- Tactical
- Composed

Expanded tone:
- Direct, not chatty
- Serious, not grim
- Capable, not flashy
- Technical, not sterile
- Refined, not luxurious

The product should read like a competent planning instrument. It can have visual character, but that character must come from disciplined structure, not from visual spectacle.

### Aesthetic Direction
Core direction:
- Operational modernism

This means:
- Dark, focused workspace optimized for long sessions
- Clean, engineered surfaces with restrained depth
- Typographic hierarchy that feels technical and intentional
- Strong spatial organization and visible state changes
- Calm contrast with a small number of meaningful accent moments

The visual language should borrow more from:
- Mission-planning software
- Briefing materials
- Technical interfaces
- Cartographic overlays
- Console-era command tools refined for modern browsers

It should borrow less from:
- Generic startup dashboards
- Neon cyberpunk UI
- Consumer productivity apps
- Gaming HUD clichés
- Glassmorphism-heavy concept shots

### Theme Strategy
Default theme direction:
- Dark-first

Reasoning:
- The product is used for sustained diagram work and dense visual inspection
- Canvas-focused workflows benefit from receding chrome and controlled contrast
- Many users will likely operate in analytical or presentation settings where a dark workspace keeps attention on the diagram

Dark mode here should not mean:
- Pure black
- Blue-purple glow aesthetics
- High-saturation accents on every control

It should mean:
- Tinted charcoal bases
- Controlled contrast
- Quiet surfaces
- One primary accent and one warm secondary signal

### Color Principles
Use color as a functional system, not decoration.

Primary color roles:
- Cool primary accent for active controls, focus, and navigational emphasis
- Warm secondary accent for alerts, structural emphasis, and selected states
- Affiliation colors remain semantically meaningful and must stay immediately distinguishable

Color priorities:
- Diagram content must remain readable above all else
- UI chrome should support the diagram rather than compete with it
- Accents should be sparse enough that they still mean something
- Neutral surfaces should be subtly tinted, never flat gray or pure black

### Typography Direction
Typography should communicate a technical tool with a modern operational edge.

Guidelines:
- Use a display family for controls, labels, section titles, and interface identity
- Use a quieter sans for body text, helper copy, and form content
- Use a mono face only for tactical labels, counters, coordinates, badges, shortcuts, and system-like readouts

Hierarchy goals:
- Headings should feel structured and assertive
- Labels should be compact and legible without feeling shouty
- Long helper text should stay calm and readable
- Numbers and short state tokens should feel precise

Typography should avoid:
- Editorial serif moods
- Friendly productivity-app softness
- Monospace overuse as a lazy “technical” signal

### Layout Direction
The app is a workspace with three major zones:
- Top command surface
- Central canvas
- Supporting side rails for palette and properties

Layout priorities:
- Preserve the canvas as the emotional center of the product
- Keep chrome visually organized but secondary to the diagram
- Make panel content feel sectional and scannable rather than boxy and repetitive
- Use asymmetry where it improves rhythm, especially in launch and summary surfaces
- Maintain strong desktop ergonomics and responsive fallback for smaller screens

The layout should feel:
- Ordered
- Layered
- Slightly architectural

It should not feel:
- Like a pile of cards
- Like a generic admin dashboard
- Like a centered marketing site

### Interaction Principles
Interactions must reinforce confidence.

Rules:
- All important controls need clear hover, active, focus, and disabled states
- Motion should clarify state change, not draw attention to itself
- Editing workflows should feel immediate and reversible
- Empty states should teach the product, not simply announce emptiness
- Dense screens should remain calm through grouping and rhythm, not through hiding critical actions

Interaction tone:
- Fast
- Crisp
- Deliberate

Avoid:
- Bouncy or elastic motion
- Showy animation loops
- Surprising transitions during focused work

### Accessibility Expectations
The design context should assume:
- Keyboard navigation matters
- Focus states must remain obvious
- Contrast should meet at least WCAG AA in all major UI text and controls
- Reduced-motion users should get a stable experience
- Dense informational surfaces should still preserve readable spacing and hierarchy

### Product-Specific Design Principles
All future design work should follow these principles:

1. Diagram First
The canvas is the product. All chrome should support faster understanding and editing of the ORBAT, not compete with it.

2. Operational, Not Theatrical
Use design to create confidence and clarity. Avoid gimmicks, neon effects, cinematic sci-fi tropes, or overtly “military” decoration.

3. Dense But Calm
The interface can support a lot of information, but it must do so with rhythm, sectional hierarchy, whitespace, and restrained color.

4. State Must Be Visible
Selections, mode changes, sharing states, unsaved work, read-only behavior, and hierarchy relationships must read instantly.

5. Briefing-Ready by Default
Users should feel that the app can move fluidly from editing to presenting to exporting without switching visual paradigms.

6. Character Through Discipline
The UI should have a memorable identity, but that identity should come from typography, spacing, palette control, and compositional rigor rather than decoration.

### Anti-References
Future work should explicitly avoid:
- Purple-blue AI gradients
- Generic SaaS bento-card repetition
- Soft consumer-app whimsy
- Gaming HUD icon overload
- Glassmorphism used as decoration everywhere
- Overly bright tactical green-on-black terminal mimicry
- Overcrowded toolbar chrome with no grouping logic

### Reference Cues
Useful inspiration categories for future design work:
- Modern command-and-control software
- Technical mapping tools
- High-quality cartographic interfaces
- Refined avionics-like panel discipline
- Briefing decks with strong typographic structure

These are cues for tone and discipline, not instructions to imitate military hardware literally.

### Quality Bar
Target quality bar:
- Flagship utility product

That means future design work should aim for:
- Strong visual identity
- Professional-grade restraint
- Reviewable, maintainable implementation
- High usability under real workload

### Technical Constraints
Derived from the repo:
- Vanilla HTML/CSS/JavaScript
- Static client-side app
- No framework migration
- No unnecessary dependency growth
- Design changes should remain performant and easy to maintain
- Existing interaction model and file structure should be preserved unless there is strong product value in changing them

### Assumptions
The following are inferred from the repository rather than confirmed directly by the creator:
- The app is primarily desktop-first
- The product’s strongest audience is professional or serious hobby analysis rather than casual use
- Dark mode is the correct primary environment
- The brand should feel operational and composed rather than consumer-friendly

If these assumptions change, this document should be updated before major design work continues.
